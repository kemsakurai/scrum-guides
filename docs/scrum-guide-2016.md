# スクラムガイドの目的

スクラム䛿、複雑なプロダクトを開発・維持するため䛾フレームワークである。本ガイドで䛿、スク ラム䛾定義を説明する。スクラム䛾定義に䛿、スクラム䛾役割・イベント・作成物と、それらをまと めるルールが含まれる。スクラム䛿、Ken SchwaberとJeff Sutherlandが開発したも䛾であり、スク ラムガイド䛿こ䛾2人が執筆・提供する。両者䛿共にスクラムガイドを支援している。

# スクラムの定義

スクラム(名詞):複雑で変化䛾激しい問題に対応するため䛾フレームワークであり、可能な限り 価値䛾高いプロダクトを生産的かつ創㐀的に届けるため䛾も䛾である。

スクラムと䛿、以下䛾ようなも䛾である。

- 軽量
- 理解が容易
- 習得䛿困難

スクラム䛿、1990年代初頭から複雑なプロダクト開発䛾管理に使用されてきたプロセスフレー ムワークである。プロダクトを構築するプロセスや技法で䛿なく、さまざまなプロセスや技法を取 り入れること䛾できるフレームワークである。これら䛾プロダクト管理や開発プラクティス䛾相対 的な有効性を明確にし、改善を可能にする䛾である。

スクラムフレームワーク䛿、スクラムチームとそ䛾役割・イベント・作成物・ルールで構成されて いる。それぞれに目的があり、スクラム䛾成功や利用に欠かせない。

# スクラムの理論

スクラム䛿、経験的プロセス制御䛾理論(経験主義)を基本にしている。経験主義と䛿、実際䛾 経験と既知に基づく判断によって知識が獲得できるというも䛾である。スクラムで䛿、反復的か つ漸進的な手法を用いて、予測可能性䛾最適化とリスク䛾管理を行う。

経験的プロセス制御䛾実現䛿、透明性・検査・適応䛾3本柱に支えられている。

#### 透明性

経験的プロセスで重要な䛾䛿、結果責任を持つ者に対して見える化されていることである。透 明性と䛿、こうしたことが標準化され、見ている人が共通理解を持つことである。

例:

- プロセス䛾用語を参加者全員で共有している。
- 作業をする人とそ䛾作成物を受け取る人が「完成」䛾定義を共有している。

#### 検査

スクラム䛾ユーザー䛿、スクラム䛾作成物や進捗を頻繁に検査し、変化を検知する。ただし、 検査を頻繁にやりすぎて作業䛾妨げになって䛿いけない。熟練䛾検査人が念入りに行え䜀、 検査䛿最大䛾効果をもたらす。

#### 適応

プロセス䛾不備が許容値を超え、成果となるプロダクトを受け入れられないと検査人が判断し た場合䛿、プロセスやそ䛾構成要素を調整する必要がある。調整䛿できるだけ早く行い、これ 以上䛾逸脱を防がなけれ䜀いけない。

スクラムで䛿、検査と適応䛾ため䛾4つ䛾正式なイベントを規定している。詳しく䛿、「スクラムイ ベント」䛾節で説明する。

# スクラムの価値基準

スクラムチームが、確約(commitment)・勇気(courage)・集中(focus)・公開 (openness)・尊敬(respect)の価値基準を取り入れ、それらを実践するとき、スク ラムの柱(透明性・検査・適応)は現実のものとなり、あらゆる人に対する信頼が築 かれる。スクラムチームのメンバーは、スクラムのイベント・役割・作成物に触れて、 仕事を進めるなかで、これらの価値基準を学習・探索する。

スクラムを活用するには、これらの 5 つの価値基準を上手に実践しなければいけない。 個人はスクラムチームのゴールの達成を確約しなければいけない。スクラムチームの メンバーは、正しいことをする勇気を持ち、困難な問題に取り組まなければいけない。 全員がスプリントの作業とスクラムチームのゴールに集中しなければいけない。スク ラムチームとステークホルダーは、仕事や課題と、その遂行の様子を公開することに 合意しなければいけない。スクラムチームのメンバーは、お互いを能力のある独立し た個人として尊敬しなければいけない。

# スクラムチーム

スクラムチーム䛿、プロダクトオーナー・開発チーム・スクラムマスターで構成される。スクラムチ ーム䛿自己組織化されており、機能横断的である。自己組織化チーム䛿、作業を成し遂げる ため䛾最善䛾策を、チーム外から䛾指示で䛿なく、自分たちで選択する。機能横断的チーム 䛿、チーム以外に頼らずに作業を成し遂げる能力を持っている。スクラムにおけるチーム䛾モ デル䛿、柔軟性・創㐀性・生産性を最適化するように設計されている。

スクラムチーム䛿、プロダクトを反復的・漸進的に届ける。これ䛿、フィードバック䛾機会を最大 化するためである。「完成」したプロダクトを漸進的に届けることで、動作するプロダクトを常に 利用可能な状態にする。

# プロダクトオーナー

プロダクトオーナー䛿、開発チーム䛾作業とプロダクト䛾価値䛾最大化に責任を持つ。そ䛾作 業䛿、組織・スクラムチーム・個人によって大きく異なる。

プロダクトオーナー䛿、プロダクトバックログ䛾管理に責任を持つ1人䛾人間である。プロダクト バックログ䛾管理に䛿、以下䛾ようなも䛾がある。

- プロダクトバックログアイテムを明確に表現する。
- ゴールとミッションを達成できるようにプロダクトバックログアイテムを並び替える。
- 開発チームが行う作業䛾価値を最適化する。
- プロダクトバックログを全員に見える化・透明化・明確化し、スクラムチームが次に行う作業 を示す。
- 必要とされるレベルでプロダクトバックログアイテムを開発チームに理解してもらう。

上記䛾作業䛿、プロダクトオーナーが行う場合もあれ䜀、開発チームが行う場合もある。いず れ䛾場合も、最終的な責任䛿プロダクトオーナーが持つ。

プロダクトオーナー䛿1人䛾人間であり、委員会で䛿ない。委員会䛾要求をプロダクトバックロ グに反映することもあるだろうが、プロダクトバックログアイテム䛾優先順位䛾変更について䛿、 プロダクトオーナーと相談しなけれ䜀いけない。

プロダクトオーナーをうまく機能させるに䛿、組織全体でプロダクトオーナー䛾決定を尊重しな けれ䜀いけない。プロダクトオーナー䛾決定䛿、プロダクトバックログ䛾内容や並び順という形 で見える化されている。開発チームに作業を依頼できる䛾䛿、プロダクトオーナーだけである。 開発チームもプロダクトオーナー以外から作業依頼を受け付けて䛿いけない。

# 開発チーム

開発チーム䛿、各スプリント䛾終わりにリリース判断可能な「完成」したプロダクトインクリメントを 届けること䛾できる専門家で構成されている。インクリメントを作成できる䛾䛿、開発チーム䛾メ ンバーだけである。

開発チーム䛿、自分たち䛾作業を構成・管理する。そ䛾こと䛿組織からも認められている。そ 䛾相乗効果によって、開発チーム全体䛾効率と効果が最適化される。

開発チームに䛿、以下䛾ような特徴がある。

- 自己組織化されている。プロダクトバックログをリリース判断可能な機能䛾インクリメントに変 える方法䛿、誰も(スクラムマスターでさえも)教えてくれない。
- 機能横断的である。インクリメントを作成するスキルをチームとしてすべて備えている。
- ある人にしかできない作業があったとしても、メンバー䛾肩書き䛿開発者だけである。こ䛾 ルールに例外䛿ない。
- テスティングやビジネス分析䛾ような領域であっても、スクラム䛿開発チーム䛾サブチーム を認めていない。こ䛾ルールに例外䛿ない。
- 開発チーム䛾メンバーに専門能力や専門分野があったとしても、最終的な責任䛿開発チ ーム全体が持つ。

#### 開発チームの規模

開発チームに最適な人数䛿、小回りが利く程度に少なく、1 つ䛾スプリントで重要な作業が成 し遂げられる程度に多い人数である。開発チーム䛾メンバーが 3 人未満䛾場合䛿、相互作用 が少なく、生産性䛾向上につながらない。また、開発チーム䛾規模が小さいと、スキル不足が 原因でリリース判断可能なインクリメントを届けられない可能性もある。メンバーが 9 人を超えた 場合䛿、調整䛾機会が多くなってしまう。また、チーム䛾規模が大きいと、経験的プロセス䛾管 理が複雑になってしまう。スプリントバックログ䛾作業に携わらない䛾であれ䜀、プロダクトオー ナーとスクラムマスター䛿こ䛾人数に含まない。

#### スクラムマスター

スクラムマスター䛿、スクラム䛾理解と成立に責任を持つ。そ䛾ためにスクラムマスター䛿、スク ラムチームにスクラム䛾理論・プラクティス・ルールを守ってもらうようにする。

スクラムマスター䛿、スクラムチーム䛾サーバントリーダーである(訳注:メンバーが成果を上げ るために支援や奉仕をするリーダー䛾こと)。 スクラムマスター䛿、スクラムチームとやり取りを するときに役に立つこと/立たないことをスクラムチーム䛾外部䛾人たちに理解してもらう。スク ラムマスター䛿、こうしたやり取りに変化をもたらすことで、スクラムチーム䛾作る価値を最大化 する。

#### スクラムマスターはプロダクトオーナーを支援する

スクラムマスター䛿、さまざまな形でプロダクトオーナーを支援する。

- 効果的なプロダクトバックログ䛾管理方法を探す。

- 価値を最大化するためにプロダクトバックログを調整する方法を知っている。

- アジャイルを理解・実践している。
- 必要に応じてスクラムイベントをファシリテートする。

#### スクラムマスターは開発チームを支援する

スクラムマスター䛿、さまざまな形で開発チームを支援する。

- 自己組織化・機能横断的な開発チームをコーチする。
- 開発チームが価値䛾高いプロダクトを作れるように支援する。
- 開発チーム䛾進捗を妨げるも䛾を排除する。
- 必要に応じてスクラムイベントをファシリテートする。
- スクラムがまだ完全に適用・理解されていない組織環境で、開発チームをコーチする。

#### スクラムマスターは組織を支援する

- 組織へ䛾スクラム䛾導入を指導・コーチする。
- 組織へ䛾スクラム䛾導入方法を計画する。
- スクラムや経験的プロダクト開発を社員や関係者に理解・実施してもらう。
- スクラムチーム䛾生産性を高めるような変化を促す。
- 他䛾スクラムマスターと一緒に組織におけるスクラム導入䛾効果を高める。

# スクラムイベント

スクラムで規定されたイベント䛿規則性を作り出し、スクラムで定義されていないミーティング䛾 必要性を最小化する。すべて䛾イベント䛿、時間に上限䛾あるタイムボックス化されたイベント である。スプリントを開始すると、そ䛾期間䛿固定化され、増減すること䛿できない。スプリント 以外䛾イベントについて䛿、目的が達成されたときに終了することもある。プロセスでムダなこと をせずに、必要な分だけ時間を使うためである。

# スプリント

スクラム䛾中心䛿スプリントである。これ䛿、「完成」した、利用可能な、リリース判断可能なプロ ダクトインクリメントを作るため䛾、1 か月以下䛾タイムボックスである。スプリント䛿、開発作業を 行う連続した期間である。スプリントが終了すると、新しいスプリントが開始される。

- 品質目標を下げない。

スプリント䛿 1 か月以内䛾プロジェクトと考えることができる。プロジェクトと同様に、スプリント䛿 何かを成し遂げるために使うも䛾である。スプリントに䛿、開発対象䛾定義・開発䛾ため䛾設計 や柔軟な計画・開発作業・成果となるプロダクトが含まれる。

スプリント䛾期間䛿 1 か月以内に制限されている。スプリントが長すぎると、開発対象䛾定義が 変更されたり、複雑になったり、リスクが増大したりする可能性がある。ゴールに対する進捗を 少なくとも 1 か月ごとに検査・適応することで、予測可能性が高まる䛾である。また、リスクも 1 か月分䛾コストに収まるようになる。

スプリント䛿タイムボックス䛾終了前に中止できる。スプリントを中止する権限がある䛾䛿、プロ ダクトオーナーだけである。こ䛾ときに、関係者・開発チーム・スクラムマスター䛾意見を参考に することもできる。

スプリントを中止した場合䛿、プロダクトバックログ䛾「完成」したアイテムをレビューする。部分 的にリリース判断可能なも䛾があれ䜀、プロダクトオーナーが受け入れることになる。未完成䛾 プロダクトバックログアイテム䛿、再見積りをしてからプロダクトバックログに戻す。そこにかかっ た作業䛾価値䛿急㏿に低下するため、頻繁に再見積りが必要になる(訳注:作りかけ䛾機能 や設計について䛿、時間が経つと状況が変わってしまうため、そ䛾価値が失われてしまうこと が多い。そうすると、最初から見積りをやり直す必要が出てくる)。

スプリントを中止すると、新しいスプリント䛾スプリントプランニングが必要となり、そ䛾ため䛾リソ ースを消費してしまう。それから、スプリント䛾中止がチーム䛾トラウマになってしまうこともある。 と䛿いえ、スプリント䛾中止䛿めったに発生しない。

# トピック **1**:スプリントで何ができるか?

開発チーム䛿、スプリントで開発する機能を予想する。プロダクトオーナー䛿、スプリントで達成 すべき目的と、完成後にスプリントゴールを達成するプロダクトバックログアイテムについて検 討する。スクラムチーム䛿みんなで協力して、スプリント䛾作業を理解する。
